  • Yesterday: why we redesigned a Business Intelligence course and what the design looks like
  • Today: the technical side — how pull requests function as assessment infrastructure
  • Relevant if you do project-based or team courses where collaboration matters

The problem: collaboration is invisible

Final deliverables hide process

Who did what, how decisions were made, and how feedback was handled is invisible in the final product

Peer interaction is difficult to audit

Retrospective reconstruction of who reviewed, commented, or contributed is unreliable

Feedback resets instead of accumulating

Each assignment starts from scratch — prior issues are not carried forward

Project-based courses are widely used in CDIO — and we care about collaboration, reasoning, and responsiveness to feedback.

But in practice, assessment still prioritizes final artifacts — reports, code, presentations.

  • The final deliverable hides most of what matters
  • Peer interaction is impossible to reconstruct after the fact
  • Feedback resets between tasks instead of building up

The issue is not lack of activity — it is lack of visibility.

Why pull requests

What a PR is

A structured proposal to change shared work: rationale, peer review, discussion, and explicit approval before merging

Why it matters for assessment

  • reasoning becomes visible
  • contribution becomes attributable
  • critique is captured
  • responses to feedback are auditable

Directly transferable

Industry-standard workflow — students practice what they will immediately use after graduation

A pull request — PR — is a structured proposal to change shared work.

It includes a rationale, gets reviewed and discussed, and is explicitly approved before merging.

  • Reasoning becomes visible in the PR description
  • Contributions are attributable — you can see who did what
  • Critique is captured in review comments
  • Responses to feedback are auditable over time

And critically — this is industry-standard practice. Students are not learning a course tool; they are learning the workflow they will use on day one.

Assessable behaviors made visible

Required practices

  • focused PRs with clear intent
  • at least two substantive reviews
  • explicit responses to all comments
  • merge only after review

Observable evidence

  • authorship and co-authorship
  • depth of critique (no LGTM or )
  • handover quality
  • documentation updates

We define specific required practices — not just “use GitHub.”

  • Focused PRs with clear intent
  • At least two substantive reviews — actionable feedback, not just approval
  • Explicit responses to all comments
  • Merge only after review

Assessment then uses observable evidence: authorship, critique depth, follow-up, handover quality.

Scaffolded progression

across courses

Information Engineering (IE)

  • 20–25 students, 6–7 small teams
  • early undergraduate
  • new repository each cycle
  • focus: learning the tools

Business Intelligence (BI)

  • 10–15 students, 2 large teams
  • assumes GitHub fluency
  • single persistent repository
  • focus: tooling supports advanced project work

This is implemented across two courses with a deliberate progression.

  • In Information Engineering: focus is learning the tools — small teams, repeated practice, new repositories each cycle
  • In Business Intelligence: students are assumed fluent — one persistent repository across the entire semester

So there is progression — from learning the workflow to using it as infrastructure for something more ambitious.

This is important: you don’t want students spending cognitive load on Git when they should be focused on the domain problem.

Feedback carry-forward

Cumulative assessment

  • unresolved PR comments remain active
  • later work is checked against prior feedback
  • capstone evaluation includes earlier issues

Effect

  • continuity across assignments
  • accountability — issues can’t be ignored
  • coherence in the final product
  • mirrors professional practice

When teams work with a single persistent repository, feedback doesn’t reset between assignments — it carries forward.

  • An unresolved PR comment stays visible in the repository
  • Later work is evaluated in light of earlier feedback
  • The capstone includes everything — students cannot simply move on

This creates accountability and continuity — much closer to how professional work actually operates.

  • Students cannot polish a submission in isolation and hand it in
  • They have to revisit and resolve — and that revision is itself assessable

Indicative evidence

Calibration

  • same dimensions, increasing expectations
  • signals what Capstone requires
  • early grading normalized to “very good”

Observed trend

  • GitHub usage improved over repeated cycles
  • teamwork & participation improved more strongly

This is a design case, not a controlled study — but the signal is consistent across cycles.

  • The GitHub usage and teamwork dimensions are shared across both courses
  • Early on, grading is deliberately lenient — students are still learning the workflow
  • We observe improvement in GitHub use and stronger improvement in teamwork and participation

Qualitatively: PRs become more focused, better documented, and more responsive over the semester.

The rubric also does something important — students see exactly what the capstone will require from week one. No surprises.

What this makes visible

Transferable principles for CDIO-aligned, project-based courses.

Visible process

Collaboration, reasoning, and iteration are observable — not just the final product

Cumulative feedback

Unresolved issues stay visible — feedback accumulates instead of resetting

Individual accountability

Contributions are attributable without fragmenting teamwork

Professional workflow

Students practice industry-standard tools in routine coursework — directly transferable

Credibility

Reproducible, documented work builds trust with industry partners

Assessment architecture

This is a design choice, not a tool choice — the structure comes first

These six principles apply beyond software and data courses — the domain matters less than the design.

  • Students often find this uncomfortable at first — reviewing peers, documenting work, responding to feedback is unfamiliar
  • But graduates say it’s where they learned the most — because the hands-on experience is real
  • Critical feedback and visible work: better to encounter them first in a course than on the job

The contribution here is an assessment architecture — not a tool recommendation.

Happy to take questions — a few that often come up:

  • Isn’t this a lot of overhead? — Yes, initially. But it decreases quickly and assessment becomes much easier and more transparent
  • What about students with no GitHub experience? — We scaffold earlier in the curriculum; in later courses we assume fluency
  • Does this scale? — It can, but requires coordination around review quality; the visibility actually helps manage larger groups
  • Is this just about GitHub? — No. GitHub is the enabler. The contribution is the assessment architecture
